# Dev-инструмент позиционирования спрайтов (только для разработки)

## Context

**Проблема.** В проекте позиции спрайтов подбираются вручную и вынесены в `define ..._POS = (x, y)` (см. `game/0_prologue/scene1.rpy` — в комментариях многократно «плейсхолдеры, подогнать по месту в игре»). Сейчас единственный способ найти координаты — угадывать числа, перезапускать, смотреть, править. Долго и неточно.

**Задача.** Дать возможность во время запущенной игры перетаскивать изображение мышью и сразу видеть готовую строку `anchor (ax, ay)` + `pos (x, y)` для вставки в `.rpy`. **Критично:** инструмент не должен попадать в продовый билд.

**Как это решается движком.** `config.developer` в проекте не задан → работает дефолт Ren'Py `"auto"`: `True` при запуске из SDK, `False` в собранном дистрибутиве. Плюс `build.classify(pattern, None)` физически исключает файлы из билда (уже применяется в `options.rpy` для `**~`, `**.bak` и т.п.). Двойная защита: весь dev-код в отдельной папке `game/dev/`, регистрация под `if config.developer`, папка исключена из сборки.

**Решение (по выбору пользователя).** Два способа запуска + полнофункциональный live-тюнер:
1. **Глобальная клавиша F9** — оверлей-тюнер поверх любой живой сцены, ничего заранее писать не надо.
2. **Dev-лейбл-песочница** — шаблон для изолированного тюнинга спрайта на чистом фоне, переход через консоль (Shift+O → `jump`).

## Архитектура инструмента

**Принцип — недеструктивный «призрак».** Тюнер не двигает реальный спрайт в дереве сцены (иначе потерялись бы rotate/матрицы времени суток). Вместо этого поверх сцены рисуется перетаскиваемая полупрозрачная копия выбранного изображения (`get_sprite_by_tag(tag)`), стартующая в текущей позиции реального спрайта. Пользователь тащит копию, читает координаты, вставляет в `define`, жмёт Shift+R — и видит, как реальный спрайт встал на место. Для точного визуала с поворотом используется dev-лейбл-песочница, где спрайт настраивается своим трансформом.

**Позиция через Drag.** Backbone — стандартный `Drag` с колбэками `dragging`/`dragged` (как в присланном пользователем примере): `drag.x, drag.y` = левый-верх копии, пишутся в `default`-переменные. Старт `drag.x/y` берётся из `get_sprite_bounds(tag)` (левый-верх bbox). Размер `w, h` — оттуда же.

**Пересчёт под якорь.** Ren'Py `pos` ставит точку якоря в (x, y). Поэтому для выбранного якоря `(ax, ay)`:
```
pos_x = drag.x + int(round(ax * w))
pos_y = drag.y + int(round(ay * h))
```
HUD показывает пару `anchor (ax, ay)` + `pos (pos_x, pos_y)` — ровно то, что вставляется в трансформ/define. Tab циклит якорь по списку `[(0.5,0.5), (0.0,0.0), (0.5,1.0), (1.0,1.0)]` (дефолт — центр). Все числа — `int` (правило проекта: float в pos движок трактует как долю экрана).

**Возможности HUD (полный live-тюнер):**
- Список показанных спрайтов (`get_showing_sprites()`) → клик выбирает; копия-призрак = его изображение.
- Drag мышью; стрелки ← ↑ ↓ → — сдвиг на 1px, Shift+стрелки — на 10px.
- Переключатель якоря (Tab).
- Строки readout: `anchor (ax,ay)`, `pos (x,y)`, `top-left (drag.x,drag.y)`, `size w×h`.
- Кнопка/клавиша «Копировать» → `CopyToClipboard("anchor (ax, ay)\npos (x, y)")`.
- «Готово»/F9 — закрыть.
- Тонкий контур bbox + крестик в точке якоря на призраке (визуальная привязка).

## Файлы

### Создать: `game/dev/dev_position_tool.rpy` (единственный новый файл кода)

Новая папка `game/dev/` — один уровень вложенности, без числового префикса (не этап повествования, по аналогии с `common/`, `libs/`, `gui/`). Всё в одном файле:

1. **`default`-переменные состояния** тюнера (dev-флаги — через `default`, не `$`/`init python`, по правилу проекта): выбранный тег, `drag.x/y`, индекс якоря, кэш `w/h`.
2. **`transform`/функции** для призрака (контур+крестик). Опереться на паттерн function-трансформов из `common/camera_fx.rpy`.
3. **`screen dev_position_tuner(initial_tag=None)`** — модальный (`modal True`, `zorder 1000`): список спрайтов, Drag-призрак, readout-панель, `key`-биндинги (стрелки/Tab/копирование), кнопки. Колбэки `dragging`/`dragged` пишут `drag.x/y` в переменные.
4. **`screen dev_hotkey_controller()`** — крошечный always-shown контроллер: `key "K_F9" action ToggleScreen("dev_position_tuner")`. Мирроринг существующего паттерна `fx_noise_screen`.
5. **`label dev_position_sandbox`** — шаблон-песочница: `scene` чистый фон → `show` целевой спрайт своим трансформом → `call screen dev_position_tuner("<tag>")` → показать полученную строку. Комментарий-инструкция: копировать под каждый спрайт, запускать через Shift+O → `jump dev_position_sandbox`. Служебные лейблы dev — не трогают сценарный текст.
6. **Регистрация под dev-флагом:**
   ```renpy
   init python:
       if config.developer:
           config.always_shown_screens.append("dev_hotkey_controller")
   ```
   Так в билде (developer=False) контроллер не регистрируется даже если файл каким-то образом попал в сборку.

Строки HUD — служебные, dev-only, файл исключён из билда и из tl-экстракта → оставить обычными строками (без `_()`), пометить комментарием, что это осознанно (не игровой/не переводимый текст).

### Изменить: `game/options.rpy` (одна строка в существующем `init python:` build-блоке)

Добавить рядом с существующими `build.classify(...)`:
```renpy
build.classify('game/dev/**', None)   # dev-инструменты — не в дистрибутив
```
Исключает и `.rpy`, и скомпилированный `.rpyc` из папки. Формат пути — как в закомментированном примере рядом (`build.classify('game/**.png', 'archive')`).

**Файлов больше не трогаем.** Ни `scene1.rpy`, ни другие сцены не редактируются (сценарный текст неприкосновенен; инструмент самодостаточен).

## Подтверждённые API (по SDK 8.5.3)

| Нужно | Что использовать | Проверено |
|---|---|---|
| Копировать в буфер | Action `CopyToClipboard(s)` | `renpy/common/00action_other.rpy:944` |
