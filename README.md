# Шаркающий человек | Shuffling Man

Жанр: Визуальная новелла / психологический хоррор.
Сеттинг: Москва, начало 2000-х

## О проекте

"Шаркающий человек" — история о распаде семьи, психическом расстройстве и границе между реальностью и галлюцинацией, рассказанная через призму ненадежного рассказчика.

### Ключевые особенности

Ненадежный рассказчик: История подается через записи Марины, но её восприятие реальности искажено психическим состоянием.

Двойственность повествования: Игрок видит "идеализированную" версию событий глазами Марины (чистая палата, ухоженные волосы, приветливый врач), которая постепенно разрушается, открывая мрачную реальность (побритая голова, камера для душевнобольных, металлический стол).

Атмосфера бытового ужаса: Хоррор строится на нарастающем психологическом давлении — от мелких семейных раздражителей до экзистенциального кошмара.

Визуальный стиль: Реалистичные бэкграунды российской хрущевки 2000-х годов с нарастающим визуальным искажением, отражающим психическое состояние героини.

## Для разработчика

### Технологический стек:

- renpy-8.5.3-sdk – движок игры
- ./game/libs/7dots.rpy – библиотека 7dots с базовыми функциями-расширениями. **Обязательна к использованию**: перед реализацией любой функциональности (аудио, трансформации, анимации, переходы, эффекты, UI, управление диалогом) сначала проверить, есть ли готовое решение в 7dots, и использовать его, не дублируя. Справочник — skill `7dots-reference` (см. CLAUDE.md)

### Конвенции

Основано на документации Ren'Py 8.5.3 (`renpy-8.5.3-sdk/doc/`). Python внутри движка — 3.12.

#### Структура файлов

- Весь код игры — в `game/`, файлы `.rpy`. Код допустимо делить на файлы, но аккуратно и осмысленно — избыточное дробление создаёт путаницу. Служебный код — `characters.rpy`, `images.rpy` (при необходимости), `libs/` — библиотеки.
- Папки внутри `game/` допустимы, но максимум один уровень вложенности (папка на главу, папка для концовок). Глубже не вкладывать.
- Имена файлов и папок — строго на английском языке, в нижнем регистре. В `.rpy`-файлах, папках и аудио слова разделяются `_`; исключение — файлы изображений, где имя строится по формату «тег атрибут» через пробел (см. «Изображения»).
- Имена файлов — избыточные: имя полностью дублирует структуру повествования, даже внутри папки главы. Файл должен однозначно идентифицироваться по одному имени, без учёта пути.
  - Сцены: `prologue_scene_1.rpy`, `chapter_1_scene_1.rpy`, `chapter_1_scene_3.rpy` (в папках `prologue/`, `chapter_1/` и т.д.).
  - Концовки: отдельная папка `endings/`, файлы `ending_1.rpy`, `ending_2.rpy`.
  - Мини-игры: в папке своей главы, имя дублирует структуру вплоть до сцены: `chapter_2/chapter_2_scene_3_minigame_piano_1.rpy`.
- Имена файлов не начинаются с `00` (зарезервировано движком) и начинаются с буквы или цифры.
- При удалении/переименовании `.rpy` обязательно удалять осиротевший `.rpyc`, иначе старый код продолжит исполняться. `*.rpyc`, `game/cache/`, `game/saves/` — в `.gitignore`.
- Отступы — только пробелы (4), единообразно во всём блоке.

#### Переменные: define / default / $

- `define` — только для констант (персонажи, transitions, transforms, пути): не сохраняется, менять после init нельзя.
- `default` — для ЛЮБОЙ переменной, меняющейся по ходу игры (флаги, счётчики). Никогда не инициализировать игровые флаги через `$` в start или `init python` — сломаются сейвы и новые игры.
- `$` — одиночные строки Python по ходу сценария; `python:` — для блоков; `init python:` — функции/классы. Временные переменные — в `python hide:`.
- Не переиспользовать одно имя под персонажа и флаг. Имена с `_` в начале зарезервированы движком. Приоритеты init — в диапазоне −999…999, у обычного кода 0+.
- Файлы, генераторы, лямбды, корутины не пикнутся в сейв — не хранить их в переменных store (только внутри `python hide` или функций).

#### Персонажи

- Все персонажи — через `define` в одном файле, короткие имена: `define m = Character("Марина", color="#...")`.
- Реплики без рассказчика — просто строкой (narration). Шаблонные вариации персонажа — через `kind=`.
- Имя героини в галлюцинациях/искажениях можно менять через `dynamic=`.

#### Изображения

- Основной способ определения — папка `game/images/` (подпапки допустимы): имя файла в нижнем регистре = имя образа. Формат имени: `тег атрибут атрибут`, напр. `marina room sad.png` → `show marina sad`.
- Теги и атрибуты — латиницей, в нижнем регистре, без слов `at`, `as`, `behind`, `onlayer`, `with`, `zorder`.
- Бэкграунды — тег `bg`: `bg kitchen day.jpg`. Сложные/составные образы — оператором `image` в `images.rpy`.
- Ассеты из `shuffling_man_assets/Черновик` копируем в `game/images/` уже с правильными именами; `.psd` в игру не кладём.
- `scene` — смена сцены (очищает слой), `show`/`hide` — спрайты, переходы — `with dissolve` и т.п.
- Анимации и трансформации — через ATL, а не Python.

#### Аудио

- Файлы — в `game/audio/`, ogg/opus предпочтительно. Имя файла в нижнем регистре без пробелов/дефисов → автоматически доступно как имя в `play music имя`.
- Каналы: `music` — фоновая музыка, `sound` — эффекты, `audio` — параллельные звуки, `voice` — озвучка.
- При повторном входе в локацию использовать `play music ... if_changed`. Затухания — `fadeout`/`fadein`.

#### Лейблы и поток управления

- Глобальные лейблы уникальны на весь проект и дублируют структуру повествования по той же схеме, что имена файлов: `chapter_1_scene_2`, `prologue_scene_1`, `ending_1`. Внутренние точки сцены — локальные лейблы `.loop`, `.after_choice`.
- `jump` — линейный переход, `call` — только когда нужен возврат; у каждого `call` в релизе должен быть `from`-клаус (авто-добавляется при сборке, опция «Add from clauses to calls»).
- Игровые циклы писать на языке Ren'Py (`while`, `menu`, `jump`), не в `python:` — иначе ломается сохранение посреди цикла.

#### GUI и стили

- Простая настройка — переменные `gui.*` в `gui.rpy`; экраны — в `screens.rpy`. Не переопределять экраны целиком без необходимости.
- Конфигурация (`config.*`) и стили меняются только в init-фазе, не по ходу игры.
- `config.name`, `config.version` — поддерживать актуальными в `options.rpy`.

#### Текст и перевод

- Игра пишется на русском (язык None). Все строки интерфейса оборачивать в `_("...")`.
- В диалогах спецсимволы `{`, `[`, `%` экранируются удвоением (`{{`, `[[`, `%%`), кавычки — `\"`.
- Не редактировать уже написанные реплики без нужды после генерации `tl/` — ломаются идентификаторы переводов. Если правка неизбежна (опечатка), сохранять идентификатор через `id`-клаус: `e "Реплика." id start_61b861a2`.

#### Мультиязычность: как писать код, чтобы переводы добавлялись легко

Диалоги переводятся автоматически (генерация `translate`-блоков в `tl/<язык>/`), поэтому правила ниже касаются в основном интерфейса, кода и мини-игр.

- Любая литеральная строка, которую увидит игрок, — в `_("...")` в месте определения: тексты экранов, кнопки, имена персонажей (`Character(_("Марина"))`), пункты настроек, `renpy.notify`, подсказки мини-игр. `_()` помечает строку для перевода, перевод происходит при отображении.
- Строки собирать целиком, не склеивать из кусков: порядок слов в других языках другой. Вместо конкатенации — подстановки: `_("Найдено предметов: [count]")`. Переведённое значение в подстановке — через флаг `!t`: `"[mood!t]"`, где `mood` определён как `_("...")`.
- Одинаковые короткие строки с разным смыслом различать тегом контекста: `_("Начать{#menu}")`, `_("Начать{#minigame}")` — отображаются одинаково, переводятся раздельно.
- Логика не зависит от отображаемых строк: сравнивать ключи/константы (`if answer == "piano_key_3"`), не переведённый текст. В сейвы и `persistent` писать ключи, не строки — сохранённый текст не перепереведётся при смене языка.
- Мини-игры: весь видимый текст — через `text`/`textbutton` со строками в `_()`; никакого текста, «запечённого» в картинки. Если без текста в графике никак — класть переведённый вариант файла в `game/tl/<язык>/<тот же путь>`, движок подхватит его автоматически. Верстать экраны с запасом по ширине: перевод бывает в 1.5–2 раза длиннее оригинала; размеры кнопок не подгонять под длину русской строки.
- Служебные лейблы (внутренние циклы мини-игр, технические переходы) помечать `hide` (`label minigame_loop hide:`), чтобы не засорять файлы переводов.
- Шрифты: не хардкодить в экранах, использовать `gui.text_font` / стили — тогда для языка достаточно `translate <язык> python: gui.text_font = "..."` и `translate <язык> style`-блоков. Выбирать шрифты с покрытием кириллицы и латиницы.
- Python внутри `translate`-блоков — без побочных эффектов: при смене языка блок исполняется заново.
- Русские UI-строки поверх встроенных английских строк движка — через `translate None strings:` (файл `game/tl/None/common.rpym` уже отвечает за системные строки).
- Генерация переводов из CLI: `renpy-8.5.3-sdk\renpy.exe shuffling_man translate english`, выгрузка для переводчиков: `... dialogue english --strings`. Переключение языка в игре — action `Language("english")` / `Language(None)` на экране настроек. Проверка — Shift+D → «Show Translation Info».

#### Проверка и запуск

- Перед коммитом прогонять линт (из корня `shuffling_man_project`): Windows — `renpy-8.5.3-sdk\renpy.exe shuffling_man lint`, Linux/macOS — `renpy-8.5.3-sdk/renpy.sh shuffling_man lint`. Линт проверяет операторы Ren'Py, но не их Python-эквиваленты — предпочитать операторы (`show`, `play`) функциям (`renpy.show()`).
- Запуск из CLI: `renpy-8.5.3-sdk\renpy.exe shuffling_man run`.
- Коммиты — осмысленные, по одной сцене/фиче; в репозиторий не коммитим сейвы, кэш, `persistent`.
